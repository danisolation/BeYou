<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| Render/start command -> seed_demo_data | Startup invocation can write demo data unless guarded by runtime mode. |
| unauthenticated login request -> auth API | External caller can attempt seeded demo credentials. |
| auth API -> session creation | A denied demo login must not cross into backend-owned session issuance. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-28-07 | Tampering | `seed_demo_data` | mitigate | Add `settings.is_production_pilot` guard before all DB writes; tests assert no demo rows in production pilot. |
| T-28-08 | Spoofing | `/api/auth/login` | mitigate | Deny `user.is_demo` when `production_pilot` or `ALLOW_DEMO_LOGIN=false`; return 403 before `create_session`. |
| T-28-09 | Elevation of Privilege | Demo admin account | mitigate | Production pilot demo login guard applies to all demo roles including admin before session creation. |
| T-28-10 | Information Disclosure | Login denial | mitigate | Use a generic production-pilot demo-account denial string; do not expose credentials, cookie values, or raw IDs. |
| T-28-11 | Repudiation | Login failures | mitigate | Call `record_login_failure` for denied demo login attempts, preserving existing auth failure tracking. |
| T-28-12 | Information Disclosure | HttpOnly session cookie | mitigate | Do not modify `create_session`, `set_session_cookie`, or frontend token behavior; backend-owned HttpOnly cookie contract remains intact. |
</threat_model>
