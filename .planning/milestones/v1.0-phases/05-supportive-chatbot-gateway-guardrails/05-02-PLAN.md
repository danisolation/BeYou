# Phase 05 Plan 02: Student Chat UI and Admin Safety Config UI

## Objective

Add student-facing chat UI and minimal admin safety config UI using existing cookie-authenticated frontend patterns, Vietnamese supportive copy, and no secret exposure.

## Requirements Covered

- CHAT-01, CHAT-03, CHAT-04, CHAT-06, ADMIN-04

## Dependencies

- Plan 05-01 backend routes and DTOs.
- Existing Phase 04 student dashboard/SOS UI patterns.

## Tasks

1. **Tests first**
   - Add `frontend/tests/phase5-chatbot-ui.test.tsx`.
   - Cover chat API helper endpoints, no token/localStorage writes, student dashboard entry, first response/non-clinical copy, high-risk SOS CTA, admin config page, and no API key rendering.

2. **Frontend API helpers**
   - Add `frontend/lib/chat-api.ts` with typed DTOs for student chat and admin config endpoints.

3. **Student UI**
   - Add `/student/chat` page with welcome, message bubbles, input, sending/error/high-risk states.
   - Add dashboard card linking to chat.
   - Link high-risk escalation to existing `/student` SOS support area.

4. **Admin UI**
   - Add admin dashboard card.
   - Add `/admin/chatbot` page with keywords/copy editor and provider status read-only display.

5. **Validation**
   - Run targeted Vitest `npm run test -- --run phase5-chatbot-ui`.
   - Run full frontend `npm run test -- --run`.

## Acceptance Criteria

- Student can open chat UI and send messages through backend helper only.
- UI never references or renders provider API key fields.
- First response and static copy avoid therapist/doctor/diagnosis claims.
- High-risk UI suggests SOS/trusted adult with supportive copy.
- Admin can save safety config without secret fields.

## Plan-Checker Self-Review

- Copy matches UI-SPEC and Phase 1 safety guide.
- Existing `apiFetch` with cookies is reused; no token storage.
- No third-party UI registry/dependency.
- Admin page manages config only, not reports or raw transcripts.
