# Phase 05 Plan 01: Backend Chat Gateway, Storage, Provider, and Guardrails

## Objective

Implement the backend-only student chatbot gateway with student-owned chat storage, provider abstraction/fallback, pre/post high-risk guardrails, metadata-only audit, and admin safety config APIs.

## Requirements Covered

- CHAT-01, CHAT-02, CHAT-03, CHAT-04, CHAT-05, CHAT-06, ADMIN-04

## Dependencies

- Phase 01 safety/audit/privacy contracts.
- Phase 02 cookie auth/admin roles.
- Phase 04 SOS source enum and escalation boundary.

## Tasks

1. **Tests first**
   - Add `backend/tests/test_phase5_chatbot_backend.py`.
   - Cover student chat via backend, first response disclaimer, no frontend/API key exposure in response, pre-LLM high-risk escalation, post-LLM safety replacement, student-only transcript reads, metadata-only audit, and admin config mutation.

2. **Domain/migration**
   - Add SQLAlchemy models and Alembic revision `20260521_0005_chatbot_gateway_guardrails.py`.
   - Extend cleanup helpers in existing tests for new FK tables.

3. **Services/schemas/provider**
   - Add chat schemas, provider abstraction, deterministic fallback, freemodel adapter.
   - Add guardrail detector using persisted safety config with safe defaults.
   - Store chat messages/signals and audit high-risk events without raw content.

4. **Routes/auth**
   - Add student chat router and admin chatbot config router.
   - Register routers in `main.py`.
   - Extend `require_permission` for `chat_thread`, `chat_transcript`, and `chatbot_safety_config`.

5. **Validation**
   - Run Alembic upgrade.
   - Run `python -m pytest tests\test_phase5_chatbot_backend.py -q`.
   - Run full backend `python -m pytest -q`.

## Acceptance Criteria

- Student can post a message and receive a Vietnamese supportive bot response through backend.
- High-risk input never reaches unrestricted provider path.
- High-risk provider output is replaced with escalation guidance.
- Admin config endpoint omits API key/secret fields.
- Transcript reads are own-student only and audited metadata-only.
- Backend tests pass.

## Plan-Checker Self-Review

- No external test calls: fallback provider default and monkeypatchable provider hook.
- No data-loss risk: additive migration only.
- No SOS duplication: chatbot suggests existing SOS but does not auto-create external notifications.
- No adult surveillance: no adult/admin raw transcript endpoint planned.
