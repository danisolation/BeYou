# Phase 1 - UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn/ui planned; not initialized because no frontend scaffold or `components.json` exists yet |
| Preset | none |
| Component library | Radix via shadcn/ui when frontend scaffold exists |
| Icon library | lucide-react |
| Font | Inter or system sans fallback; Vietnamese text must render clearly |

Implementation note: when the Next.js frontend is created, initialize shadcn/ui before building Phase 1 UI surfaces. Use only official shadcn components unless a later phase completes registry safety vetting.

## Spacing Scale

Declared values must be multiples of 4:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge spacing |
| sm | 8px | Compact element spacing |
| md | 16px | Default card padding and form spacing |
| lg | 24px | Section padding on mobile |
| xl | 32px | Desktop section padding and page gaps |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level spacing |

Exceptions:
- Minimum touch target: 44px height/width for buttons, links, checkboxes, and mobile controls.
- SOS action button in later phases: minimum 56px height, never visually tiny.
- Privacy notice cards: 16px mobile padding, 24px desktop padding.

## Typography

Use exactly these type sizes and no additional phase-specific sizes.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 14px | 400 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.2 |

Rules:
- Use weight 400 for normal explanatory text.
- Use weight 600 for headings, key labels, and primary action emphasis.
- Avoid clinical or legal-heavy tone in student-facing text.
- Vietnamese copy must be short, warm, and readable for THPT students.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#F8FCFF` | App background, onboarding surface, calm page background |
| Secondary (30%) | `#EAF7F3` | Cards, privacy notice panels, dashboard/profile notice links |
| Accent (10%) | `#2CA58D` | Primary CTA, acknowledgement checkbox active state, supportive success indicators |
| Warning | `#F59E0B` | Mild warning, "Can chu y", privacy exception notices |
| Destructive | `#DC2626` | SOS/high-risk only; destructive actions only |

Accent reserved for:
- Primary CTA: privacy acknowledgement.
- Active consent/acknowledgement checkbox.
- Supportive "safe/understood" status indicators.
- Links to revisit privacy notice from dashboard/profile.

Color rules:
- Red is reserved only for SOS, high-risk safety states, or destructive confirmation.
- Soft orange is used for mild warning and "needs attention" states.
- Adult/admin surfaces must not use alarming risk heatmaps or leaderboard-like colors.
- Demo environment indicators must be visible but not alarming: use blue badge/banner styling.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | `Xác nhận quyền riêng tư` |
| Empty state heading | `Chưa có nội dung an toàn để hiển thị` |
| Empty state body | `Khi có thông tin mới, BeYou sẽ hiển thị ở đây để em biết ai có thể xem dữ liệu nào.` |
| Error state | `Chưa tải được nội dung quyền riêng tư. Hãy thử lại hoặc báo với người phụ trách demo.` |
| Destructive confirmation | None in Phase 1. For SOS/high-impact confirmation use: `Gửi tín hiệu để người lớn tin cậy biết em cần hỗ trợ?` |

Required student-facing copy surfaces:

### Privacy notice placement

- Show after login during onboarding.
- Keep persistent links from dashboard and profile.
- Link label: `Ai có thể xem thông tin của em?`

### Privacy notice content structure

Use these headings:

1. `Ai thấy gì?`
2. `Khi nào cần báo người lớn?`
3. `Dữ liệu demo và dữ liệu thật khác nhau thế nào?`
4. `BeYou hỗ trợ em như thế nào?`

### Required privacy message

Use this meaning exactly:

`Thông tin của em được giữ riêng tư trong BeYou. Một số người lớn tin cậy chỉ xem phần tóm tắt cần thiết để hỗ trợ em, không xem mọi câu trả lời riêng tư. Nếu có dấu hiệu em có thể không an toàn, BeYou có thể cần báo cho người lớn tin cậy để giúp em kịp thời.`

Do not promise absolute confidentiality.

### Self-check labels

Use these exact labels:

- `On dinh`
- `Can chu y`
- `Nen tim ho tro`
- `Can ho tro som`

These are support states, not diagnoses.

### Chatbot disclaimer

`BeYou không thay thế chuyên gia tư vấn hay bác sĩ. Mình có thể lắng nghe và giúp em nghĩ về bước an toàn tiếp theo.`

### SOS copy

Primary SOS text:

`Gửi tín hiệu để người lớn tin cậy biết em cần hỗ trợ.`

Confirmation copy:

`Em có muốn gửi tín hiệu hỗ trợ ngay bây giờ không? Người lớn tin cậy được liên kết với em sẽ nhận thông báo trong BeYou.`

Tone:
- Serious, calm, not panic-inducing.
- No gamification.
- No promise of automatic emergency-service contact.

### Demo-data banner

`Đang xem dữ liệu demo - không phải hồ sơ học sinh thật.`

Demo metadata badge:

`Demo`

### Adult-view tone

Adult views must use support-oriented language:
- Use: `học sinh cần được quan tâm`, `gợi ý hỗ trợ`, `tóm tắt được phép xem`
- Do not use: `giám sát`, `xếp hạng rủi ro`, `học sinh nguy hiểm`, `theo dõi bí mật`

## Interaction Contract

### Privacy onboarding

- Present as a full page or modal after login before dashboard access.
- Student must acknowledge with a checkbox before continuing.
- Checkbox label: `Em đã đọc và hiểu ai có thể xem thông tin của em.`
- Continue button remains disabled until checkbox is checked.
- After acknowledgement, dashboard/profile links still allow review.

### Dashboard/profile privacy link

- Always visible in student dashboard/profile.
- Link text: `Ai có thể xem thông tin của em?`
- Opens the same privacy notice content without requiring re-acknowledgement.

### Demo separation

- Demo mode must show a persistent top banner.
- Demo records must show a `Demo` badge near the record title or metadata.
- Never rely only on color to distinguish demo data.

### Visibility matrix UI

When displaying policy/permission rules, use plain-language tables:

| Resource | Student | Teacher | Parent | Admin |
|----------|---------|---------|--------|-------|
| Self-check raw answers | Own only | Not shown by default | Not shown by default | Restricted support/admin purpose only |
| Self-check summary | Own | Linked students only | Linked child only | Restricted support/admin purpose only |
| Chat transcript | Own only by default | Not shown by default | Not shown by default | Not shown by default |
| Chat safety signal | Own summary | Safety summary only when needed | Safety summary only when needed | Audit/safety metadata only |
| SOS alert | Own | Linked/assigned students only | Linked child only | Operational metadata only |

### Audit policy UI

Audit event displays must show metadata only:
- Actor
- Action
- Resource type
- Timestamp
- Reason/status
- Metadata summary

Never show raw chat content, passwords, tokens, or full self-check answers in audit UI.
